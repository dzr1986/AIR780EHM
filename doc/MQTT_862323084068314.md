### 5.2 `2002` — 低功耗 → `1002`

**进入 rest**

```json
{"dataType":"2002","lowPowerMode":"enter"}
```

**退出 rest**

```json
{"dataType":"2002","lowPowerMode":"exit"}
```

| 条件 | 行为 |
|------|------|
| enter（**常电 → rest**） | 断 T3x、进 rest，**MQTT 保持**；成功后发 **1002**（`source=enter`） |
| enter（**已在 rest**） | 收到并处理（日志 `d2in`、`lp+ mqtt_2002`），**不发 1002**；仅周期 **1003** 仍上报 |
| exit（**rest → 常电**） | 唤醒 T3x、出 rest；成功后发 **1002**（`lowPowerMode=exit`，`reason=mqtt_2002` 等） |
| exit（**已在 normal**） | 收到并处理（日志 `d2out`），**不发 1002**（无状态切换） |
| enter + USB 已插入 | 忽略进 rest（`block_4g_rest_when_usb`），**无 1002、无 1004**（静默；日志 `d2u`） |
| 2002 字段非法 | **无上行**（日志 `d2?`）；判态用 **1003** |

> **备注 — 已在 rest 时重复 2002 enter 为何无 1002**
>
> 固件仅在 `low_power_mode` **从 0 变为 1** 时调用 `publishRest`（`user/app.lua` → `setLowPowerMode`）。若开机已因 USB 拔出等进入 rest（如 `reason=usb_remove`），再发 `{"dataType":"2002","lowPowerMode":"enter"}` 时：
>
> - 串口可见：`d2in` → `lp+ mqtt_2002` → 周期 `p3 …/status`（1003）
> - **不会出现** `p2 …/rest`（1002）
> - _alive 日志中 `lp1` 表示已在 rest，属正常
>
> **平台判态**：以 **1003.lowPowerMode**（`normal` / `rest`）为准，勿因未收到 1002 认为 enter 失败。
>
> **2002 lowPowerMode enter/exit 均走 `…/rest` 主题 1002**，不走 `event`（1004 仅用于 **2004** 控制应答及 OTA）。
>
> **联调若要收到 enter 1002**，建议顺序：
>
> 1. `{"dataType":"2002","lowPowerMode":"exit"}` → 收 **1002**（`lowPowerMode=exit`）
> 2. `{"dataType":"2003"}` → 1003 中 `lowPowerMode` 应为 `normal`
> 3. `{"dataType":"2002","lowPowerMode":"enter"}` → 收 **1002**（`lowPowerMode=enter`，`reason=mqtt_2002`，`source=enter`）

> **备注 — rest 下 2002 exit 预期日志**
>
> 设备已在 rest 时下发 `{"dataType":"2002","lowPowerMode":"exit"}`，串口/MQTT 典型顺序：
>
> ```text
> d2out
> mdl {"dataType":"2002","lowPowerMode":"exit"}
> lp- mqtt_2002
> p2 /panshi/app/862323084068314/rest  exit mqtt_2002   ← 1002，lowPowerMode=exit
> … lp0 … mmqtt-ok
> p3 /panshi/app/862323084068314/status  … bat 68 interval 30
> ```
>
> - **1002 在 `onExitLowPower` 状态切换成功后发出**（与 enter 对称），T3x 唤醒在其后异步进行。
> - **exit 不发 1001**；当前态以 **1003.lowPowerMode** 为准。
> - **已在 normal 时重复 exit**：无 `lp-`、无 **1002**、不重复唤醒 T3x。

**1002 enter 示例** → `/panshi/app/862323084068314/rest`

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1002",
  "lowPowerMode": "enter",
  "reason": "usb_remove",
  "source": "enter",
  "time": "2026-06-11 22:00:00"
}
```

**1002 exit 示例** → `/panshi/app/862323084068314/rest`

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1002",
  "lowPowerMode": "exit",
  "reason": "mqtt_2002",
  "time": "2026-06-11 23:01:39"

```

**reason 常见值**：`usb_remove` · `battery` · `mqtt_2002` · `at` · `usb_insert` · `boot_no_usb` · `unknown`

### 5.3 `2003` — 状态查询 / 配置周期 → `1003`

**立即查询**

```json
{"dataType":"2003"}
```

**配置 1003 上报间隔（秒）**（Cat.1 `net_mqtt.handleDownlink2003` → `setStatusIntervalSec`）

```json
{"dataType":"2003","interval":30}
```

| 项 | 说明 |
|----|------|
| 合法范围 | **10～86400** 秒（`clampIv`） |
| 持久化 | `/mqtt_status_cfg.json`（`status_interval_sec`） |
| 运行时 | 同步 `APP_RUNTIME.low_power_interval_sec`、`LOW_POWER_CFG.rest_mqtt_interval_sec` |
| 周期定时 | `MQTT_STATUS_INTERVAL_CHANGED` 触发重算等待时间 |
| 设备日志 | 成功 `d3c 30` + `d3ok 30`；非法 `d3?` |

**仅查询**（不改 interval）：`{"dataType":"2003"}` → 立即 1003，日志 `d3q`。

**应答** → `/panshi/app/862323084068314/status`

配置 interval 时，1003 额外带 `ret` / `message`：

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1003",
  "usbInserted": 0,
  "charging": 1,
  "remainPower": "67",
  "batteryMv": "3808",
  "lowPowerMode": "rest",
  "interval": 30,
  "ret": 0,
  "message": "ok",
  "time": "2026-06-11 22:00:00"
}
```

| 字段 | 说明 |
|------|------|
| `usbInserted` | GPIO27 外壳 USB，`0/1` |
| `charging` | GPIO17 充电态，`0/1` |
| `remainPower` | 电量 % |
| `batteryMv` | 电芯 mV |
| `lowPowerMode` | `normal` 常电 / `rest` 低功耗 |
| `interval` | 当前 1003 周期（秒），配置后应等于下发值 |
| `ret` / `message` | 仅带 `interval` 的 2003 时有；`0`/`ok` 表示配置成功 |

### 5.9 `2012` — 设备开录 → `1012`

平台下发 **2012**，由 **4G** 开启录像会话并唤醒 T3x；**T3x 不解析 MQTT**。与 **2011↔1011** 对称：个位对齐 **2012↔1012**，上行主题均为 **`event`**。完整时序见 [mqtt_2012_1012_flow.md](./mqtt_2012_1012_flow.md)。

```json
{"dataType":"2012","messageId":"start-001"}
```

可选字段：`action`、`uploadMode`、`quality`、`videoMaxDurationSec`（与 2010 媒体字段一致；省略则用当前 2010 配置）。

| 条件 | 说明 |
|------|------|
| 未在录像 | `pir_ctrl.session.recording=0` |
| `startOnCloud=1` | 2010 `recordPolicy`（可用 2010 query 确认 `start_cloud=1`） |
| 订阅 | `/panshi/app/862323084068314/event`（**1012 与 1011 同主题**） |

受理或拒绝后 **立即 1004** `action=pir_start`；受理另发 **1012**；T3x 写盘后可能 **1010** `t3x_active`（`pir`）。

| 情况 | 上行 |
|------|------|
| 受理成功 | **1004** `ret=0` + **1012** `source=4g` |
| T3x 写盘 | 可选 **1010** `t3x_active`（`pir`） |
| 拒绝 | **1004** `ret=-1`，`message=denied|busy|suspended|…` |

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1012",
  "reason": "device",
  "source": "4g",
  "action": "video",
  "uploadMode": "auto",
  "quality": "high",
  "recording": 1,
  "time": "2026-06-12 12:00:00"
}
```

### 5.10 `2011` — 平台停录（TF 卡）→ `1011`

平台下发 **2011**，由 **4G** 结束录像会话并停 **TF 卡** 写盘；**T3x 不解析 MQTT**，通过串口协作停录。完整时序见 [mqtt_2011_1011_flow.md](./mqtt_2011_1011_flow.md)。

```json
{"dataType":"2011","messageId":"stop-001"}
```

| 条件 | 说明 |
|------|------|
| 正在录像 | `pir_ctrl.session.recording=1` |
| `stopOnCloud=1` | 2010 `recordPolicy`（可用 2010 query 确认） |
| 订阅 | `/panshi/app/862323084068314/event`（**1011 在此主题**） |

受理或拒绝后 **立即 1004** `action=pir_stop`；停 TF 卡录完成后另发 **1011**（写盘时可能延迟数秒）。

| 情况 | 上行 |
|------|------|
| 受理 | **1004** `ret=0`；随后 **1011**（见下行） |
| 拒绝 | **1004** `ret=-1`，`message=not_recording` / `stop_on_cloud_denied` |
| T3x 未在写盘 | **1011** `reason=device`，`source=4g` |
| T3x 在写盘 | 先等 `AT+RECORD=0` → **1011** `source=t3x`；超时 **15s** 补发 `source=4g`（`11fb`） |

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1011",
  "reason": "device",
  "source": "t3x",
  "uploadMode": "auto",
  "quality": "high",
  "time": "2026-06-12 12:00:00"
}
```

`reason` 也可能是 T3x 原样（如 `done`）；`source=t3x` 表示 IPC 已停录确认。

### 5.11 `2020` — 编码查询 → `1020`

平台下发 **2020**，由 **4G** 经串口向 T3x 查询编码参数（`AT+VENC?` / `AT+AUDIO?`）；**T3x 不解析 MQTT**。个位对齐 **2020↔1020**，上行主题 **`encode`**。完整字段与排障见 [REMOTE_ENCODE_CONFIG.md](./REMOTE_ENCODE_CONFIG.md)。

```json
{"dataType":"2020","messageId":"q-all"}
```

```json
{"dataType":"2020","camera":0,"stream":0,"messageId":"q-cam0-main"}
```

```json
{"dataType":"2020","scope":"audio","camera":0,"messageId":"q-audio0"}
```

| 条件 | 说明 |
|------|------|
| 处理侧 | **仅 Cat.1**；IPC 走 UART |
| T3x 状态 | rest/休眠时需 **4G 唤醒上电** 后再查 |
| 订阅 | `/panshi/app/862323084068314/encode` |
| 超时 | 可选 `timeoutMs`；默认约 **12000** ms |

**无即时 1004**；应答为 **1020**（`reply=1`，`ret`/`message`/`body`）。

| 情况 | 上行 |
|------|------|
| 查询成功 | **1020** `ret=0`，`body.video` 或 `body.audio` |
| T3x 不可用 | **1020** `ret=-1`，`message=t3x_unavailable` 等 |
| 超时 / 串口忙 | **1020** `ret=-1`，`message=timeout` / `busy` |

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1020",
  "reply": 1,

  "ret": 0,
  "message": "ok",

    "video": [
      {

        "stream": 0,
        "enable": 1,
        "width": 1920,
        "height": 1080,
        "bitrate": 1200,
        "framerate": 25,
        "rcmode": 2,
        "encoder": 4
      }
    ]
  },
  "time": "2026-06-12 12:00:00"
}
```

### 5.12 `2021` — 编码设置 → `1021`

平台下发 **2021**，由 **4G** 合并当前配置后经 UART 写入 T3x（`AT+VENCSET=` / `AT+AUDIOSET=`）。个位对齐 **2021↔1021**，上行主题 **`encode`**。

**视频（主码流示例）**

```json
{
  "dataType": "2021",
  "camera": 0,
  "stream": 0,
  "width": 1920,
  "height": 1080,
  "bitrate": 1200,
  "framerate": 25,
  "rcmode": 2,
  "encoder": 4,
  "messageId": "set-v0"
}
```

**仅改码率（通常热更新，`needReboot=0`）**

```json
{
  "dataType": "2021",
  "camera": 0,
  "stream": 0,
  "bitrate": 800,
  "messageId": "set-br"
}
```

**音频**

```json
{
  "dataType": "2021",
  "scope": "audio",
  "camera": 0,
  "enable": 1,
  "encoder": 4,
  "samplerate": 8000,
  "bitwidth": 16,
  "volume": 80,
  "gain": 28,
  "messageId": "set-audio0"
}
```

| 条件 | 说明 |
|------|------|
| `camera` | `0`–`3`（与 T3x 路数一致） |
| `stream` | 视频：`0` 主 / `1` 子 |
| `scope` | 缺省视频；`audio` 为音频 |
| 订阅 | `/panshi/app/862323084068314/encode` |

| 变更类型 | 固件行为 | `needReboot` |
|----------|----------|--------------|
| 仅 **bitrate** | 热更新码率 + 写 ini | 通常 **0** |
| 分辨率 / 帧率 / encoder 等 | 写 ini + T3x **重启** | **1** |

```json
{
  "deviceNo": "862323084068314",
  "dataType": "1021",
  "reply": 1,
  "messageId": "set-v0",
  "ret": 0,
  "message": "ok",
  "needReboot": 1,
  "time": "2026-06-12 12:05:00"
}
```

与 **2010 `quality`** 无关：`quality` 是 PIR 业务档位，不是 `syscfg.ini` 编码分辨率。

## 6. 联调步骤（MQTTX）

1. 连接 Broker（§1 参数），ClientId 用 `platform-test-001`。
2. **Subscribe**：`/panshi/app/862323084068314/#`，QoS 1。
3. **Publish** 状态查询：
   ```json
   {"dataType":"2003"}
   ```
   Topic：`/panshi/device/862323084068314/`
4. 应在 `…/status` 收到 **1003**（含 `lowPowerMode`、`remainPower` 等）。
5. 可选：Publish `{"dataType":"2005"}` → 收 **1005**；`{"dataType":"2010","action":"query"}` → 收 **1010**。
6. 编码：Publish `{"dataType":"2020","messageId":"q1"}` → 在 `…/encode` 收 **1020**（T3x 需已唤醒，见 §5.11）。
7. **测 2002 enter → 1002**：须先 **exit** 出 rest（收 **1002 exit**），再 enter（见 §5.2）；已在 rest/normal 时重复 enter/exit **无 1002**。

### 6.1 本机典型启动日志对照

| 日志 | 含义 |
|------|------|
| `I/user.nm ok` | MQTT 连接成功 |
| `I/user.nm rc23` | rest 下 conack 补发 **1002+1003** |
| `I/user.nm p2 …/rest` | 发布 **1002** |
| `I/user.nm p3 …/status` | 发布 **1003** |
| `I/user.app mAon` | MQTT 常连模式已启动 |
